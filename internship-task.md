# Задание на стажировку

## 1. Описание предметной области

Приложение для просмотра меню и бронирования еды в буфете. Бронировать еду могут только зарегистрированные пользователи. 

Администратор может составлять меню на день, неделю из уже занесенных в базу блюд. Блюда разбиты на категории (салат, горячее, гарнир, супы и т.д.). Меню на каждый день содержит определенное количество доступных порций. Вся история ежедневного меню, бронирований, должна сохраняться для построения статистических отчетов.

## 2. Функциональные требования

### 2.1 Функционал пользователей

1. Незалогиненный пользователь имеет функционал регистрации на сайте с указанием минимального набора пользовательских данных.
2. Любой пользователь, вне зависимости от факта входа на сайт имеет следующий набор функций:
    * Просмотр меню на текущий день. Для каждой позиции в меню должен  быть предусмотрен индикатор наличия порций.
    * Просмотр планируемого на неделю меню.
3. Залогиненный пользователь должен иметь следующий набор функций:
    * Просмотр и редактирование пользовательских данных
    * Просмотр и редактирование предпочитаемых блюд (выбор блюд происходит из списка, заведенного через панель администратора)
    * Бронирование блюд. Бронь действительна в течение 45 минут (подумать над тем как можно реализовать валидацию по времени). Бронировать возможно только те блюда, которое есть в наличии на момент бронирования. После бронирования происходит временный декремент оставшегося количества порций по каждому из забронированного блюда. Результатом бронирования является случайный код, по которому администратор может подтвердить факт покупки, в таком случае бронь считается подтвержденной и декремент оставшегося количества порций становится перманентным. 
    * “Мне не хватило”, кнопка, действительная на блюдах, которых больше нет в наличии, для построения отчетов.
    * Пользователь может оставить обратную связь от своего имени, либо анонимно.

### 2.2 Функционал администратора
1. В системе по умолчанию создается SA пользователь, обладающий максимальным количеством прав. SA пользователь может создавать администраторов сайта.
2. Администратор имеет следующий набор функций:
    * Создание блюд в конкретной категории (список категорий захардкоден). Блюдо имеет цену, вес, описание
    * Формирование меню на завтрашний день с указанием количества порций по каждому блюду. Формирование меню на неделю без указания количество порций
    * Просмотр статистики: в какие дни больше всего бронирований, наиболее популярные блюда, предпочитаемые блюда пользователей; блюда, которых обычно не хватает
    * Просмотр обратной связи
    * Подтверждение бронирования по коду

## 3 Технологические требования
Проект должен быть построен с учетом следующих технологий, практик:
* ASP.Net MVC, ASP.Net Core Web API, .Net Standard 2.0 (Class library)
* Функционал пользователя должен быть реализован в виде REST API (.Net 5) и MVC клиента к нему (.Net 5). Работа с БД должна быть реализована с помощью Entity Framework (Code First, Data Annotation + Fluent API)
* Функционал администратора должен быть реализован в виде классического MVC приложения с использованием Razor, вся работа с БД через ADO.Net
* MSSQL 
* Asp.Net Identity
* Для логирования в случае ASP.Net Core приложения использовать Serilog, в случае ASP.NET MVC — log4net
